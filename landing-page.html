What It Is - "The Shipping Container"

The M365 Agent SDK is the bridge that turns a raw AI agent into a deployable "app" that can live inside Microsoft Teams, Copilot, or Outlook. It ensures agents are "Agent 365 Compliant."

Think of it as: The packaging that makes your agent fit onto the Microsoft 365 truck and ensures it can be tracked by IT.

Standardized Packaging

What Gets Packaged

  • ‚Ä¢ Agent logic (from Framework/Foundry)
  • ‚Ä¢ Manifest file (capabilities, permissions)
  • ‚Ä¢ Configuration (channel-specific settings)
  • ‚Ä¢ Dependencies and resources

Output

  • ‚Ä¢ .zip file ready for deployment
  • ‚Ä¢ Agent Store submission package
  • ‚Ä¢ Custom app installation bundle

Telemetry Emission (Agent 365 Requirement)

Automatic Signals

  • ‚Ä¢ User ID: Who used the agent
  • ‚Ä¢ Token usage: Cost tracking
  • ‚Ä¢ Latency: Performance monitoring
  • ‚Ä¢ Tool calls: What the agent accessed
  • ‚Ä¢ Error rates: Reliability metrics

Why This Matters: Agent 365 requires this data for fleet observability. Without SDK telemetry, agents are "invisible" to IT.

Foundry IQ

PREVIEW KNOWLEDGE

The managed knowledge layer unifying M365 documents (Work IQ), Enterprise Data in Fabric (Fabric IQ), and public web data into a single knowledge graph. Agentic RAG with 36% higher relevance vs standard implementations.

36% higher relevance vs standard RAG implementations
Agentic retrieval: Dynamic search strategy based on query type
Work IQ (M365) + Fabric IQ (Enterprise data) connectors

What It Is - "Agentic RAG Engine"

Foundry IQ is the managed knowledge layer that sits underneath your agent fleet. It unifies disparate data sources into a single, normalized "knowledge graph" endpoint.

Instead of building custom ETL pipelines, vector stores, and retrieval logic, Foundry IQ abstracts this into a managed service that agents can query naturally.

Why It Matters

  • ‚Ä¢ 36% Higher Relevance: Microsoft benchmarks claim 36% boost in response accuracy
  • ‚Ä¢ Agentic Retrieval: Dynamically decides how to search (Keyword, Vector, Hybrid) based on query
  • ‚Ä¢ Code Reduction: Eliminates ~40% of custom RAG boilerplate (chunking, embedding, re-ranking)

Architecture Components

Work IQ

Native connector to M365 Graph (Teams, SharePoint, Outlook, OneDrive). Respects user permissions‚Äîagents can't retrieve data users couldn't access manually.

Fabric IQ

Native connector to Microsoft Fabric (OneLake, SQL, Dataverse). Query structured enterprise data without complex SQL wrappers.

Automated Indexing

Handles chunking, embedding, and re-ranking automatically. Optimizes for specific data types (sliding windows for documents, row-based for data).

Implementation Workflow

Use Case: "What was the financial impact of Project X mentioned in last week's emails?"

  1. 1. Agent receives query
  2. 2. Agent delegates retrieval to Foundry IQ
  3. 3. Foundry IQ formulates search strategy ("Search Emails for Project X" + "Query SQL for Financials")
  4. 4. Foundry IQ retrieves, re-ranks, and synthesizes context
  5. 5. Agent generates response based on high-fidelity context

Strategic Value

üí° Enterprise Impact

Solves the "fragmented context" problem that makes most enterprise agents ineffective. Grounds agents in truth, dramatically reducing hallucinations when answering questions based on internal company data.

Model Router

GA ROUTING

Smart traffic controller that sits between your application and model providers. Optimizes token spend, prevents vendor lock-in, and provides automatic failover. Route to GPT-5, Claude 4.5 Sonnet, Llama 3, and more.

Dynamic routing based on prompt complexity and rules
Automatic failover for resiliency and high availability
3 optimization modes: Quality, Cost, Latency

What It Is - "Smart Traffic Controller"

The Model Router is a unified API endpoint that sits between your agent application and raw model providers. Instead of hard-coding calls to specific models, your application calls the Router.

The Router analyzes prompt complexity and routing rules to select the optimal model in real-time. Swapping models becomes a configuration change, not a code rewrite.

Why It Matters

  • ‚Ä¢ Dynamic Routing: Analyzes prompt complexity to select optimal model in real-time
  • ‚Ä¢ Lock-In Prevention: Decouples application logic from specific model versions
  • ‚Ä¢ Resiliency: Automatic failover if primary provider experiences latency or outages

Routing Modes

üéØ Optimize for Quality

Goal: Best possible answer, regardless of cost

Behavior: Routes complex reasoning to GPT-5 or Claude 4.5 Sonnet

üí∞ Optimize for Cost

Goal: Lowest token spend for acceptable quality

Behavior: Routes simple tasks to GPT-4o-mini, Phi-4, or Llama 3

‚ö° Optimize for Latency

Goal: Fastest Time-to-First-Token (TTFT)

Behavior: Prioritizes fastest available model endpoint

MaaS Catalog (Models as a Service)

Route traffic to third-party models as first-class citizens:

  • ‚Ä¢ OpenAI: GPT-5, o1
  • ‚Ä¢ Anthropic: Claude 4.5 Sonnet
  • ‚Ä¢ Meta: Llama 3
  • ‚Ä¢ Mistral: Large
  • ‚Ä¢ Cohere: Command R+
  • ‚Ä¢ Microsoft: Phi family

Strategic Value

üí° Immediate Adoption

Adopt immediately for all production workloads. Easiest way to reduce costs (offload simple tasks to cheaper models) and ensure reliability (failover protection). Configuration change vs code rewrite.
